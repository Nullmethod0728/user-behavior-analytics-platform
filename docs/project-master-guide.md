## 1. 这个项目到底在解决什么问题

### 先理解一个真实场景

你每天打开手机上的**应用商店**（比如华为应用市场、OPPO 软件商店）。

你在应用商店里的每一个动作都会被记录下来：
- 打开应用商店 → 记录一次"浏览首页"
- 搜索"微信" → 记录一次"搜索"
- 点进微信详情页 → 记录一次"点击"
- 点下载按钮 → 记录一次"下载"
- 下载完点安装 → 记录一次"安装"

OPPO 有 **5 亿台设备**，每台设备每天产生几十条这样的记录。每天就是**几百亿条日志**。

### 问题来了

| 谁想知道什么 | 要求 |
|------------|------|
| 运营总监 | "现在实时在线多少人？" | 1 秒内出结果 |
| 产品经理 | "昨天微信的下载转化率是多少？" | 每天看一次 |
| 广告团队 | "曝光到点击的转化正常吗？突然跌了要告警" | 实时检测异常 |
| 数据分析师 | "上周新用户的 7 日留存率趋势" | 离线跑报表 |

**核心矛盾**：同样一批数据，不同的人需要不同的处理方式。

### 如果你的项目是 OPPO 的真实系统

你的项目做的事就是：**把几百亿条用户日志，变成不同角色需要的答案**。

## 3. 逐层深度剖析

### 第一层：日志生成器（Python）

**它在做什么？**

造假数据。因为 OPPO 不会把真实的 5 亿用户日志给你做个人项目。

**它是怎么造的？**

```
前提准备:
  - 5000 个虚拟用户 (user_000001 ~ user_005000)
  - 3000 台虚拟设备 (OPPO Find X6 Pro、vivo X100 等)
  - 30 个中国城市
  - 25 个知名 App (微信、抖音、王者荣耀...)

每造一条日志:
  1. 从 5000 人中随机选一个 → 模拟真实用户反复活跃
  2. 按概率选事件类型 (60% 浏览, 20% 点击, 5% 下载...)
  3. 从设备池随机分配一台设备
  4. 生成当前时间戳
  5. 根据事件类型生成对应的 properties:
     - 下载事件 → 带 app_name、app_id
     - 搜索事件 → 带 keyword
     - 支付事件 → 带 order_id、金额
  6. 打包成 JSON → 发给 Kafka
```

**面试要点**：

"我模拟了 8 种事件类型，比例是按真实应用商店设计的。事件类型不同，携带的扩展属性也不同——比如下载事件会带 app_id 和 app_name，搜索事件带 keyword。这些设计参考了真实的埋点规范。"

---

### 第二层：Kafka 消息队列

**它解决什么问题？**

解耦 + 缓冲。用一个最简单的场景说明：

```
没有 Kafka：
  日志生成器 → Flink        ← 如果 Flink 挂了，日志丢了
  日志生成器 → Spark        ← 如果要加第三个消费者，要改生成器代码

有 Kafka：
  日志生成器 → Kafka → Flink    ← Flink 挂了？Kafka 存着数据等你恢复
                     → Spark   ← 加新消费者？直接订阅 Topic 即可
                     → 新系统   ← 生成器一行代码不用改
```

**Topic**

Kafka 的核心概念。类比：一个公告栏。

```
Topic "user-behavior-log" = 专门张贴"原始日志"的公告栏
Topic "user-behavior-dwd" = 专门张贴"清洗后数据"的公告栏
Topic "real-time-metrics" = 专门张贴"计算结果"的公告栏
```

我们建了 3 个 Topic，因为不同形态的数据放在一起会乱。就像快递分拣：原始包裹、已分拣、待签收各放一个筐。

**Partition**

一个 Topic 可以切成多个分区。类比：超市收银台。

```
1 个分区 = 1 个收银台 → 顾客排队久
3 个分区 = 3 个收银台 → 同时处理 3 条队伍，效率 x3
```

我们设了 3 个分区，因为开发环境 3 个够了。生产环境会设几十到上百个。

**面试要点**：

"Kafka 在这个项目里承担数据总线的角色。所有日志先进 Kafka，Flink 和 Spark 各取所需。这样生成器不需要知道下游有谁，下游也不用担心上游数据格式变化。这是微服务架构里消息队列的标准用法。"

### 第三层 A：Flink 实时计算

**它解决什么？** 毫秒到秒级的计算。数据一边来，一边算。

**和普通程序的区别**：

```
普通 Python 程序:
  while True:
      data = kafka.poll()   # 取一条
      pv += 1               # 累加
      print(pv)              # 输出
  # 问题：程序崩了，pv 归零，白算

Flink:
  data_stream = env.fromSource(kafka)   # 数据流
  result = data_stream
      .window(5分钟)
      .aggregate(pv_uv计算逻辑)
      .sink(输出到Redis)
  # 好处：程序崩了，从 Checkpoint 恢复，PV 不丢
```

**我们实现的 4 个作业**：

| 作业 | 窗口类型 | 输出 | 面试考点 |
|------|---------|------|---------|
| RealtimePVUV | Tumbling 5分钟 | Redis | 窗口、去重 (HyperLogLog) |
| DownloadStats | Sliding 5分钟/1分钟 | ClickHouse | 滑动窗口 vs 滚动窗口 |
| FunnelAnalysis | Tumbling 10分钟 | ClickHouse | 复杂事件处理 |
| AnomalyDetector | Tumbling 5分钟 + 前后对比 | 控制台 | 状态管理、告警逻辑 |

**Tumbling Window 和 Sliding Window 的区别**：

```
Tumbling (滚动):               Sliding (滑动):
[00-05) [05-10) [10-15)       [00-05)
                               [01-06)
不重叠，每条数据只属于一个窗口     [02-07)
                               ...
                               有重叠，一条数据属于多个窗口
```

实时 PV/UV 用 Tumbling —— 只要"每 5 分钟"的总量。
下载排行用 Sliding —— 要"最近 5 分钟"的排行，每分钟更新一次。

**面试要点**：

"Flink 的核心优势是状态管理和 Checkpoint 机制。窗口计算依赖的状态（比如这 5 分钟的 PV 累积值）存在 RocksDB 里，每 60 秒做一次 Checkpoint 到持久化存储。故障恢复时从最近的 Checkpoint 重放，保证 exactly-once 语义。"

### 第三层 B：Spark 离线计算

**它解决什么？** 小时到天级的计算。读全量历史数据，算完后写结果。

**和 Flink 的分工**：

| | Flink | Spark |
|------|-------|-------|
| 什么时候跑 | 数据一来就处理 | 每天凌晨（或按需触发） |
| 处理范围 | 当前窗口内的小批量 | 昨天一整天的全量数据 |
| 场景 | "现在在线多少人" | "昨天留存率多少" |
| 原因 | 实时性要求高 | 需要全量扫描才能算准 |

**为什么不能用 Flink 算留存？**

留存计算需要知道"一个新用户在 N 天后是否还活跃"。如果你 7 月 1 日来了一个新用户，你需要在 7 月 8 日判断他是否活跃——Flink 要记住 7 天的状态，这对内存压力很大。Spark 可以一次性扫描 7 月 1 日到 7 月 8 日的所有数据，直接 JOIN 完事。

**我们实现的 5 个作业**：

```
ODS → DWD 清洗:    去重、校验、补全（洗菜）
DWD → DWS 汇总:    按天聚合一整天的指标（炒菜）
留存计算:          用户 JOIN 自己 N 天后的数据
漏斗分析:          按步骤统计独立用户数，算转化率
用户画像:          给每个用户打标签（活跃度、偏好、类型）
```

**面试要点**：

"离线作业我用了 Spark SQL 的 DataFrame API。留存计算的核心是用户表自连接——基准日的用户 LEFT JOIN N 天后活跃的用户，COUNT 交集的用户数除以基准日用户数就是留存率。这里有一个优化点：如果用户量很大，可以用 Bitmap 或 HyperLogLog 近似替代精确 JOIN。"

### 第四层：存储层（ClickHouse + Redis）

**为什么是两种数据库，不是一个？**

因为它们擅长的任务完全不同：

| | ClickHouse | Redis |
|------|------------|-------|
| 类型 | OLAP（分析型） | 内存缓存 |
| 强项 | 亿级数据秒级聚合 | 毫秒级单值读写 |
| 弱项 | 不适合频繁单条更新 | 不适合存大量历史数据 |
| 类比 | 档案馆（容量大、检索快） | 便签纸（随手写随手看） |
| 我们用它存 | 所有历史指标 | 当前这一秒的实时数字 |

**在项目里的具体使用**：

Redis 的使用方式：
```
Key: analytics:realtime:metrics
Value: {"metric":"pv_uv","pv":128000,"uv":45000,"time":"2026-07-10T14:30:00"}
TTL: 60秒 ← 60秒后自动过期删除，因为 Dashboard 只关心最新值
```

ClickHouse 的使用方式：
```
表: ods_user_behavior_log (原始日志)
  - 30 天自动删除
  - 按天分区 → 查某天数据只扫当天文件
  
表: dws_daily_metrics (天级汇总)
  - 365 天保留
  - 每天凌晨 Spark 写入一行
  - Dashboard 趋势图查的就是这张表
```

**面试要点**：

"ClickHouse 和 Redis 是互补关系，不是替代关系。Redis 做热数据缓存（实时翻牌器，60 秒过期），ClickHouse 做冷数据查询（历史趋势，保留 90 天）。这跟工业界的设计一样——Redis 挡在前面扛高并发读取，ClickHouse 在后面支撑复杂分析查询。"

### 第五层：Spring Boot API

**它解决什么？** 前端不能直接连数据库。原因：

1. **安全**：直接暴露 ClickHouse 的 IP 和密码给浏览器 → 灾难
2. **解耦**：前端说我要"留存率"，不应该关心数据在 ClickHouse 还是 MySQL 还是 Redis
3. **数据处理**：可以在这层做格式转换、聚合、缓存

**我们暴露的接口**：

```
GET /api/v1/realtime/metrics           → 读 Redis → 实时 PV/UV
GET /api/v1/analytics/retention?date=  → 读 ClickHouse → 留存率
GET /api/v1/analytics/funnel?date=     → 读 ClickHouse → 漏斗数据
GET /api/v1/analytics/daily?from=&to=  → 读 ClickHouse → 天级趋势
GET /api/v1/analytics/top-downloads    → 读 ClickHouse → 下载排行
```

**面试要点**：

"API 层做了数据源的分离——实时数据走 Redis（毫秒级返回），历史数据走 ClickHouse 的 JDBC 查询。前端完全不用感知底层有两个异构数据源。这体现了 BFF（Backend For Frontend）模式的思想。"

## 4. 每个技术选型的原因

| 如果用别的 | 会有什么问题 |
|-----------|------------|
| 不用 Kafka，直连 → | 如果 Flink 处理慢了，日志生成器卡住；下游增加消费者要改生成器代码 |
| 不用 Flink，用 Spark Streaming → | Spark Streaming 是"微批处理"，延迟秒级；Flink 是真正的"逐条处理"，延迟毫秒级 |
| 不用 ClickHouse，用 MySQL → | 查一亿条数据的 PV，MySQL 要几分钟，ClickHouse 要 0.1 秒 |
| 不用 Redis，全查 ClickHouse → | 每秒几千次 Dashboard 轮询全打到 ClickHouse，浪费计算资源 |
| 不用 Spring Boot，用 Flask → | 可以，但 Java 生态在国内大数据岗位更主流，写上去更对胃 |

## 5. 数据是怎么流动的

### 完整链路（一句话版）

> Python 生成日志 → Kafka 缓冲 → Flink 实时算（写 Redis + ClickHouse）+ Spark 凌晨算（写 ClickHouse）→ Spring Boot 统一查询 → ECharts 大屏展示

### 关键节点的时间线

```
14:00:00  一条日志诞生（Python 生成器）
14:00:01  抵达 Kafka
14:00:02  Flink 消费到，开始窗口计算
14:05:00  窗口触发，PV/UV 结果写入 Redis
14:05:01  Dashboard 轮询 API，翻牌器数字跳动
────────  一天过去了 ────────
次日 02:00  Spark 启动，读取昨天全量 ODS 数据
```
